# 龙虾投资助手自我迭代机制

## 目标

让龙虾投资助手定期检查自己的回答质量，发现信息真实性、来源准确性、投资合规、分析深度和输出体验问题，并生成可人工审核的增强建议。

核心原则：

- 使用三角色隔离：任务执行者负责回答；独立 Reviewer 负责找问题；优化建议者负责基于 Review 结果评分并提出增强方案。
- 执行者不得给自己评分，Reviewer 不得采纳执行者自评，优化建议者不得跳过 Review 直接凭感觉改配置。
- 只允许“独立审计”和“提出修改建议”，不允许无人确认自动修改线上配置。
- 任何迭代都不能放宽信息真实性、来源准确性、不喊单、不承诺收益、风险提示等底线。
- 迭代目标是提高准确性、可追溯性和分析质量，不是提高激进程度。
- 自我迭代必须以“发现问题”为导向，不以证明自己正确为目标。
- 专业性提升必须可观察：来源更准确、财务/估值/风险框架更完整、结论边界更清楚、同类错误复发更少。

## 文件结构

建议在助手配置中维护以下文件：

```text
lobster-assistant/
├── SOUL.md                         # 行为准则
├── WORKFLOW.md                     # 日常工作流程
├── skills/
│   ├── a-share-market-news.md       # A股核心要闻整理
│   ├── a-share-stock-analysis.md    # 个股分析
│   ├── task-runner.md                # 任务执行者
│   ├── independent-review.md         # 独立审计者
│   └── optimization-advisor.md       # 评分与优化建议者
├── evals/
│   ├── test_cases.md                # 固定评测题
│   └── scorecard.md                 # 评分表
└── iterations/
    └── YYYY-MM-DD-proposal.md       # 每次迭代建议
```

## 运行频率

### 冷启动期

冷启动期定义：

- 助手上线后的前 14 天。
- 或最近 50 条股票/投资回答中仍出现中高风险问题。
- 或新增模型、数据源、Skill、工作流后的 7 天观察期。

冷启动期执行频率：

- 每日轻量自检：每天 1 次，检查最近 5-10 条回答，重点看来源、日期、股票代码、公告主体、合规风险。
- 每 3 天深度迭代：每 3 天 1 次，检查最近 20-30 条回答，并运行固定评测题，输出 Prompt/Skill 修改建议。
- 每周回归测试：每周 1 次，全量运行固定评测题，检查最近几次迭代是否引入副作用。

### 稳定期

连续 2 周满足以下条件后，才允许进入稳定期：

- 没有高风险问题。
- 个股分析平均分 >= 90。
- 财经要闻平均分 >= 90。
- 来源准确性平均分 >= 22/25。
- 没有用户反馈“来源不准、日期错误、编造消息、喊单”。

稳定期执行频率：

- 轻量自检：每 3 天 1 次。
- 深度迭代：每周 1 次。
- 固定评测：每 2 周 1 次，或每次修改配置后立即运行。

### 重大变更

新增模型、数据源、Skill 或工作流后，必须立即跑一次固定评测集，并进入至少 7 天观察期。

## 三角色职责

### 角色 A：任务执行者

职责：

- 按 SOUL、WORKFLOW 和具体 Skill 完成用户任务。
- 输出答案时保留事实、推断、风险和来源。
- 产出可供审计的证据包：使用了哪些来源、哪些结论依赖哪些来源、哪些信息无法确认。

禁止：

- 不得给自己的回答打分。
- 不得声明“本回答已通过审计”。
- 不得在无来源情况下补全事实。

### 角色 B：独立 Reviewer

职责：

- 只审计任务执行者的原始回答、来源链接和证据包。
- 采用反方视角找问题，不负责重写答案。
- 输出问题清单、证据核验结果、风险等级、P0/P1/P2 分类。

隔离要求：

- 不读取执行者自评。
- 不接受“看起来合理”作为通过依据。
- 对无法打开、无法支撑结论、时间不匹配的来源必须判为问题。

### 角色 C：评分与优化建议者

职责：

- 只基于 Reviewer 的审计报告、评分表和历史复发记录进行评分。
- 判断问题根因：规则缺失、规则冲突、执行失败、数据源不足、模型能力不足、用户输入不足。
- 生成可人工审核的 Prompt/Skill/流程增强提案。

禁止：

- 不得替 Reviewer 重新审计事实。
- 不得跳过 Review 报告直接给高分。
- 不得提出会弱化真实性、来源、合规、风险提示的改动。

## 固定流程

1. 收集样本
   - 最近 N 条股票/投资相关回答。
   - 用户负反馈或追问较多的回答。
   - 涉及个股、行情、公告、政策、资金流的高风险回答优先。
   - 必须包含随机样本、低分样本、用户追问样本、合规压力样本，禁止只挑选表现好的样本。

   样本配比建议：
   - 40% 随机真实回答。
   - 25% 个股/公告/资金流等高风险回答。
   - 20% 用户负反馈、追问、纠错样本。
   - 15% 固定评测和对抗样本。

2. 任务执行
   - 角色 A 完成用户任务。
   - 输出正文之外，生成或保留证据包。
   - 证据包至少包含：来源链接、数据时间、支撑事实、无法确认的信息、关键推断链路。

3. 独立审计
   - 角色 B 逐条核验事实是否有来源。
   - 检查来源是否准确支撑结论。
   - 检查是否混淆日期、证券简称、股票代码、公告主体。
   - 检查是否出现荐股、喊单、收益承诺。
   - 检查是否解释了涨跌逻辑，而不是只堆新闻。
   - 检查是否体现专业分析框架：基本面、估值、资金面、催化、风险、反向假设。
   - 检查是否把短线情绪、中期业绩、长期产业逻辑混在一起。

4. 评分与根因分析
   - 角色 C 基于 Reviewer 报告评分，不直接相信执行者自述。
   - 按问题类型统计频率。
   - 标记高风险问题。
   - 判断是个案失误，还是需要修改 Prompt/Skill。
   - 记录同类问题是否复发；同一问题连续 2 次出现，必须升级为配置问题。

5. 生成变更提案
   - 角色 C 写清要改哪个文件。
   - 给出可直接替换的规则片段。
   - 说明修改原因和潜在副作用。
   - 明确是否建议人工合入。
   - 每个提案必须绑定至少 1 个回归测试题。
   - 每个提案必须说明“如何判断改进成功”。

6. 人工确认
   - 用户或管理员确认后，才允许修改 SOUL、WORKFLOW、Skill。
   - 未确认时，只保存提案，不生效。

7. 回归验证
   - 合入任何配置变更后，必须运行固定评测题中相关子集。
   - 如果回归测试引入新的高风险问题，必须回滚本次提案或重新生成修订提案。
   - 回归通过不代表进入稳定期；稳定期仍需满足连续 2 周无高风险问题。

## V1 自动化闭环

日常推荐链路不执行完整三角色 review。V1 自我迭代只消费运行态产物和失败记录：

- `runtime/recommendation-runs/<YYYY-MM-DD>/<session-id>/<run-id>.json`
- `runtime/feedback-log.jsonl`
- `scripts/validate_run.sh` 的失败结果

失败类型固定为：

| failure_type | 含义 |
|---|---|
| `sector_state_stale` | 板块状态过期仍被使用 |
| `sector_selection_error` | 板块选择与状态或证据不匹配 |
| `source_missing` | 关键事实缺来源 |
| `source_mismatch` | 来源不能支撑结论 |
| `extract_error` | 原始值到整理值抽取/单位/口径错误 |
| `stale_data` | 旧数据被当作新数据 |
| `price_scope_error` | 价格时间和口径错误 |
| `prediction_mislabeled` | 预测值未标注或混成真实数据 |
| `overheat_ignored` | 热门方向过热风险未降级 |
| `low_activity_overrated` | 低活跃方向被错误给高关注 |
| `unsafe_instruction` | 出现无条件交易指令、仓位或收益承诺 |

周度处理：

1. 汇总 `feedback-log.jsonl` 和 `validate_run.sh` 失败结果。
2. 统计 top failure types。
3. 只对高频或 P0/P1 问题生成规则修改提案。
4. 每个提案必须绑定失败样本、修改文件和新增/更新的校验规则。
5. 人工确认后再修改 skill/reference；日常链路不自动改规则。

## 旧版单角色流程废弃

以下做法禁止使用：

- 执行助手自己给自己评分。
- 执行助手自己总结“本轮无问题”。
- 评分者没有 Review 报告就直接打分。
- 只基于最终答案外观判断专业性。
- 用执行者自己的解释抵消来源错误、日期错误或主体错误。

## 问题分级与升级

### P0 立即阻断

出现以下任一情况，本轮结论必须判定为不可通过，并触发人工介入：

- 编造公告、财务数据、行情数据、政策内容。
- 股票代码、公司主体、交易所错误。
- 来源链接无法支撑核心结论。
- 给出明确买入/卖出/仓位指令。
- 承诺收益或使用“必涨、稳赚、翻倍”等表达。

### P1 必须修复

- 关键事实缺少来源。
- 未标注数据时间或公告日期。
- 个股分析缺少主要风险。
- 只讲利好，不讲反向风险。
- 同一类中风险问题连续 2 次出现。

### P2 持续优化

- 输出不够精简。
- 结构不够稳定。
- 分析维度不完整。
- 术语解释不够清楚。

## 专业性提升目标

每轮深度迭代至少检查以下专业能力是否提升：

- 事实核验：是否优先使用官方公告、交易所、巨潮、监管机构和可追溯财经源。
- 财务分析：是否覆盖营收、利润、现金流、毛利率、负债、同比/环比等关键指标。
- 估值分析：是否说明 PE/PB/PS/市值分位或同行对比，不能只说“估值高/低”。
- 产业逻辑：是否说明公司处在产业链哪个环节，催化如何传导到收入或利润。
- 资金行为：是否区分成交放大、主力资金、北向、龙虎榜、融资融券等不同信号。
- 反向假设：是否说明什么情况会使当前判断失效。
- 结论边界：是否明确短线、中期、长期逻辑的适用范围。

## 禁止自动修改的内容

以下规则只能增强，不能删除或弱化：

- 必须联网核验高时效金融信息。
- 关键事实必须有来源链接。
- 不得编造、不得引用未确认传闻。
- 不承诺收益，不喊单，不给明确买卖指令。
- 输出必须包含风险提示。
- 个股分析必须区分事实、推断和观点。
